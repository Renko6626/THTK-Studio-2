# 02 — TH16 子弹虚拟机模型:为什么一颗子弹不是简单 struct

> **对象**:TH16《鬼形兽》`th16.exe`,imagebase `0x400000`。日期 2026-06-09。
> 这篇是**机制/概念解释**(配合 `01-core-engine.md` 的偏移/opcode/handler 速查表读)。所有字段偏移、循环边界、
> 状态块均**已一手反编译实证**(✅),并与社区 etEx 表交叉验证(见 `01` §8);本文只解释它们**如何组合成一套 VM**。
> 仅 TH16。

## 0. 一句话

**一颗子弹 ≈ 一个微型脚本 actor**:它在自己体内携带 ①一段**字节码程序**、②一个"**当前激活了哪些效果**"的**位场**、③**每个效果各自的私有状态块**。多个 `etEx` 效果可以**同时挂在一颗弹上并发推进**(加速 + 反弹 + 环绕…各跑各的),所以它**不是** `{坐标, 速度, 精灵}` 那种扁平 struct,而是一个 **`0x1478` 字节的 VM 上下文**。

## 1. 朴素弹幕引擎 vs TH16 的做法

```
朴素做法(很多 STG):                TH16 的做法:
struct Bullet {                      struct Bullet {                // 0x1478 字节
  float x, y;                          float x,y,z, vx,vy,vz;       // 运动学
  float vx, vy;          ──────►       float speed, angle, radius;
  int sprite;                          // —— VM 控制 ——
};                                      u8   bytecode[0x318];        // +0xc88:程序(≤18 条指令)
// 每帧 x+=vx;y+=vy 完事                 int  pc;                     // +0xc60:程序计数器
                                        u32  active_fx;              // +0xc68:激活效果位场(EX_ 位)
                                        // —— 每个效果一块私有状态 ——
                                        FxState speedup;             // +0xfa0
                                        FxState accel;               // +0xfe8
                                        FxState bounce;              // +0x10d4
                                        FxState veladd;              // +0x1228
                                        ... (十余块,见 §4)
                                        SavedState saved;            // +0x1314/18/1c
                                      };
```

朴素弹只能"匀速直线"。TH16 的弹能在生命周期里**自己跑脚本**:加速、转向、到点、反弹、环绕、变精灵、变大小、播音、再生子弹/激光/敌人……而且这些能**叠加并发**。

## 2. 三层结构(都在弹对象体内)

### 层 1:程序(bytecode)— `obj+0xc88`,`obj+0xc60`=PC ✅
- 弹自带一段指令流,**步长 `0x2c`(11 dword)/ 指令**,**恰好 18 条**(spawn 时从发射器描述符拷 `0xc6` dword = `0x318` 字节 = 18×0x2c)。
- 每条指令:`word[8]`=**opcode**(其整数值 = 它要点亮的 `+0xc68` 位,见 `01` §3/§8)、`word[9]`=**wait 门**、`word[0..7]`=参数(角/速/数量/模式…)。
- **执行器** = `bullet_vm_exec`(`0x413860`):`while (PC < 0x12)` 顺序执行,直到撞 wait 门停。

### 层 2:激活位场 — `obj+0xc68` ✅
- 32 位,**每位 = 一个 etEx 效果是否正在生效**(`0x1`=EX_SPEEDUP、`0x4`=EX_ACCEL、`0x40`=EX_BOUNCE…全表见 `01` §8)。
- "行为类"opcode 执行时做 `c68 |= opcode`(opcode 整数就是那个位);效果跑完由 handler 自己 `c68 &= ~bit` 清掉。
- **多位同时为 1 = 多个效果并发**。

### 层 3:每效果私有状态块 ✅
- 每个"持续型"效果在弹体内有一块**固定偏移的私有 scratch**(计时器、目标值、插值器内部态)。彼此**不重叠**,所以能并发互不干扰(见 §4 内存图)。
- opcode 执行时**初始化**这块(写目标/时限);对应的 handler 每帧**推进**这块。

## 3. 生命周期:VM 与 handler 如何交织 ✅

```
[发射器侧] etNew/etAim/etCount/etSpeed/etAngle + 多个 etEx
            └─ 编译进"发射器效果池"(≤18 槽,etEx 的 index 寻址)
                                  │ etOn 触发 → bullet_pool_spawn(0x412cb0)
                                  ▼
[弹诞生] 把效果池拷进 obj+0xc88(18 条指令),设 PC=描述符[0xde],c68=描述符[0xdb]
                                  │
        ┌─────────────────────────┴── 每帧 bullet_tick(0x411e70) ──────────────────────────┐
        │ (a) bullet_vm_exec(0x413860):从 PC 跑字节码,直到 wait 门                          │
        │      · 行为 opcode → c68|=位 + 填该效果状态块(角/速参数在此用阈值规则解析)        │
        │      · 一次性 opcode(置精灵/播音/spawn 子弹·激光·敌人/快照…)→ 立即做, PC++       │
        │      · EX_GOTO(0x10000)跳转 / EX_WAIT·EX_DELAY 计时 → 控制节奏                     │
        │ (b) 遍历 c68 的每个置位 → 调对应 bullet_beh_*(0x414ec0…):推进该效果状态块         │
        │      · 多个效果在同一帧各推进各的(并发协程);到时限各自 c68&=~位 自停            │
        │ (c) 位置积分 obj+0xc20 += dt·obj+0xc2c …                                          │
        │ (d) bullet_vs_player_collide(0x4124b0)→ 命中判定;离屏裁剪                         │
        └────────────────────────────────────────────────────────────────────────────────┘
```

**关键点**:
- **VM(层1)是"设置者"**:把 etEx 翻译成"打开某效果 + 写它的参数"。它一帧能跑多条,直到 wait 门让出。
- **handler(层2+3)是"执行者"**:每帧把所有激活效果各推进一步。
- 二者**异步**:VM 可以先一次性挂上 EX_ACCEL + EX_BOUNCE + EX_WRAP 三个效果(三位都置 1、三块状态都填好),之后多帧里三个 handler 并发跑;也可以用 EX_GOTO/EX_WAIT 在中途再改。

## 4. 内存图:弹对象作为 VM 上下文(✅ 偏移实证)

| 区段 | 偏移 | 内容 |
| --- | --- | --- |
| 运动学 | `+0xc20/24/28`,`+0xc2c/30/34`,`+0xc38`,`+0xc3c`,`+0xc40/44` | 坐标 / 速度 / 速率 / 角 / 半径 |
| **VM 控制** | `+0xc60` PC,`+0xc64` GOTO 计数,`+0xc68` 激活位场,`+0xc72` 状态,`+0xc84` 渲染层 | |
| **程序** | `+0xc88 .. +0x1474`(`0x318` 字节 = 18×`0x2c`) | 字节码(每条 = 一个 etEx) |
| **效果状态块**(每个持续效果一块,互不重叠) | | |
| EX_SPEEDUP `0x1` | `+0xfa0` | |
| EX_ACCEL `0x4` | `+0xfe8`(目标速 `+0xffc` / 角 `+0x1000` / 时限 `+0x101c`) | |
| EX_ANGLE_ACCEL `0x8` | `+0x1030`(速率 `+0x1044` / 角率 `+0x1048` / 时限 `+0x1064`) | |
| EX_STEP `0x10` | `+0x1078`(目标速 `+0x108c` / 目标角 `+0x1090` / `+0x10ac..b8`) | |
| EX_BOUNCE `0x40` | `+0x10d4`(边界 `+0x10dc/e0` / walls `+0x1100` / 时限 `+0x10f8`) | |
| EX_DELETE/计数 `0x100000`/混合 → `+0x558` | | |
| EX_OFFSCREEN `0x100` | `+0x12b8`(倒计时 `+0x12c0`) | |
| EX_WRAP `0x1000` | `+0x1184`(walls `+0x118c` / count `+0x1188`) | |
| EX_REACT/EX_DELAY `0x4000000` | `+0x1348`(倒计时 `+0x134c`) | |
| EX_MOVE `0x20000` | 目标 `+0x1208` / 3D 插值器 `+0x1394` 块 | |
| EX_VELADD `0x80000` | `+0x1228`(速度向量 `+0x1250` / 时限 `+0x125c`) | |
| EX_VELTIME `0x200000` | `+0x1270`(每步增速 `+0x1284` / 目标角 `+0x1288` / 时限 `+0x12a4`) | |
| EX_SIZE `0x400000` | `+0x13ec`(`+0x1414/18` / 尺寸标量 `+0x141c`) | |
| EX_WAIT `0x80000000` | `+0x1108` | |
| **EX_SAVE 快照** | `+0x131c` 位 / `+0x1318` 角 / `+0x1314` 速 | 供 EX_STEP c=2/3 回读 |
| 动画/精灵 | `+0x1450/54`,`+0x1464/68`,`+0x1474` 类型 idx | |

> 这张图就是答案:弹体内为**每一种效果都预留了一块状态**,这样任意子集的效果可以同时激活。这是典型的**"宽 struct + 位场调度"**的脚本 actor 设计,不是窄数据记录。

## 5. 一个具体例子(把机制走一遍)

发射器脚本:
```
etAngle(0, 0.0, 0); etSpeed(0, 2.0, 2.0);
etEx(0, 0, EX_ACCEL,  60, 0, 0.05, NEGF);   // 效果槽0:60 帧内加速,保持当前角
etEx(0, 1, EX_BOUNCE, 3,  BOUNCE_UDLR, 0, 0);// 效果槽1:最多反弹 3 次,四壁
etOn(0);
```
诞生的每颗弹:
1. spawn 把这 2 条 etEx 编译进 `+0xc88`(指令0=EX_ACCEL、指令1=EX_BOUNCE),PC=0。
2. 第 1 帧 `bullet_vm_exec`:跑指令0 → `c68|=0x4`、填 EX_ACCEL 块(`+0xffc`=0.05、`+0x101c`=60、角=当前角);跑指令1 → `c68|=0x40`、填 EX_BOUNCE 块(`+0x10f8`=3、walls=UDLR);遇程序尾/ wait 门停。**此刻 c68 = 0x44(两位)**。
3. 之后每帧 `bullet_tick`:`bullet_beh_accel_vec`(0x4) 与 `bullet_beh_wall_bounce`(0x40)**并发**跑——一个每帧加速度积分,一个检测撞墙就反射角+镜像位置。60 帧后 accel 自清 `0x4`;弹反弹 3 次后 bounce 自清 `0x40`。
4. 全程位置积分 + 命中判定照常。

→ 同一颗弹**同时**"加速"且"反弹",各有独立计时与状态。换成扁平 struct 根本表达不了。

## 6. 设计含义(给 IDE / 移植)

- **"子弹类型"= 一段 etEx 程序**,不是一个枚举。IDE 若要可视化/编辑弹,得把 `+0xc88` 当**指令列表**(≤18 条),每条一个 EX_ + 参数,而非平铺字段。
- **发射器效果池(≤18 槽)** 与弹的 18 指令一一对应;`etEx(id, index, ...)` 的 `index` 就是槽号。
- 运行时一颗弹的"当前行为"要看 `+0xc68` 位场(可能多位),不能只看速度向量。

## 7. 设计评价与历史定位

> **可信度分层**:§7.1(架构评价)基于本工作区**一手实证**的数据/结构 → ✅。§7.2(后作沿用)有本次逆向旁证 → 🟡。§7.3(品类对比)是**行业概览判断,非逆向证据**,知识截止 2026-01,**未反编译过 TH16 以外任何作品** → ⚙️ 仅供参考、待验。

### 7.1 这是什么取舍 ✅
这套 = ZUN Windows 期(TH06 2002 起、TH10/11 前后定型)的招牌**三 VM 架构**:**ECL(玩法脚本)→ 每弹 etama 效果 VM(本文档这套)→ ANM(显示脚本)**。核心是一笔**用内存换灵活性 + 确定性**的工程交易:

| 维度 | 实测/结构(✅) | 评价 |
| --- | --- | --- |
| **可组合** | 一颗弹 = ≤18 条 etEx 程序,从 ~28 个固定 `EX_` 词汇拼行为 | 改弹幕**不改引擎**;设计师在 ECL 表达,不碰 C++。最强项。 |
| **并发效果** | `+0xc68` 位场 + 每效果私有状态槽 | 任意效果子集同时跑,组合优雅。 |
| **确定性优先** | 固定池、无每帧分配、固定迭代序,接死 16 位 PRNG 回放(`../shared/th16-engine-math.md`) | 为**逐帧 bit-exact 回放**服务——是格式硬约束,非洁癖。 |
| **代价:胖对象** | 每弹 `0x1478`≈5.2KB(每种效果都预留槽,多数常空);`2×2001` 槽 ≈ **21MB 弹池** | 这就是"为何不是简单 struct"的账单:**拿内存换组合自由**。 |
| **代价:封顶** | ~28 硬编码效果 + 每弹 18 指令上限 | 超出需改引擎;脚本层不能无限扩展。 |
| **代价:每弹解释器** | 跑 bytecode + N handler/帧 | 比 `pos+=vel` 贵,但 wait 门 + 2001 上限兜底,60fps 够用。 |

**定性**:典型的"**宽 struct + 位场调度**"脚本 actor;不是现代数据导向(DOD)的精简弹(DOD 会用 SoA + 只给激活效果开数组)。在 2002 年:实现简单、回放好对、表达力强——非常合理的取舍。

### 7.2 后作是否沿用 🟡(本次旁证)
本研究交叉验证时发现:`../ecl/ECL-info.md` 那张"7 档角度阶梯"**在 TH16 不存在**(常量缺失),更像 **TH17(2019)/TH18(2021)** 的行为(见 `01` §8.1)。→ 这套效果 VM **没被替换,而是逐版本扩词汇/加分支地继承**(thcrap 覆盖到最新作亦佐证)。**仅旁证,未实测后作。**

### 7.3 品类对比 ⚙️(行业判断,非逆向,待验)
- **理念**(声明式/脚本化/可组合弹幕)= **当代主流**:东方同人、Danmakufu(ph3)、各路独立 STG 皆然。这层"在且更盛"。
- **具体实现**(每弹**内嵌字节码解释器** + 为每效果**预留槽**的位场)= **偏 ZUN 血脉的特征**。2015 后 Unity/Godot/自研引擎的弹幕地狱更多走 **ECS / 数据导向 / GPU 实例化弹池**(几万弹更省),模式编写用 Lua/自定义 DSL 脚本层,而非"每弹一段 bytecode + 每效果留 struct 槽"。→ 这套**具体形态**是 ZUN 谱系的活化石,品类里非通用。

### 7.4 ★ 开放验证项(把 7.2/7.3 从判断变实证)
- 拿 **TH17/TH18/TH19 的 exe**,按本工作区同套锚点复核:`obj+0xc68` 风格的效果位场是否仍在?`bullet_vm_exec` 同构函数 + opcode↔EX_ 映射是否扩展(尤其那 7 档角度阶梯是否真在 TH17+ 出现)?池结构是否仍是固定胖槽。
- 纪律:**勿外推、逐版本验**;在做之前 7.2/7.3 一律按上面的可信度标注,别写成定论。
