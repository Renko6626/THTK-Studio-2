## 0. 结论速览

这是一套**敌弹 / 弹幕(danmaku)对象池引擎**——ZUN "etama" 式:一个**预分配大池**装轻量弹对象,每颗弹**自带一小段运动字节码程序**(VM)+ 一个**每帧行为位场**驱动加速/转向/反弹等运动。**身份 = 敌弹**(✅,见 §2 的伤害+池架构双重证据);其字节码能 spawn 子弹/敌人,故更准确叫"脚本化弹对象",比"哑弹"能力强(社区 etama 对名在能力层面是近似,身份层面准确)。

**★ 易混淆**:**弹管理器 `DAT_004a6dac`** vs **敌人管理器 `DAT_004a6dc0`**(只差一位十六进制!)。弹对象 `0x1478` 字节、池内嵌;敌人对象 `0x574c`(~22KB)、独立 `operator_new`。

## 1. 一刀链 + 三个核心函数 ✅

从 README §A 锚点 `0x4439e0`(玩家碰撞)反推:

```
任务调度器(优先级 0x1c 回调 thunk @0x412c50,先查暂停标志 DAT_004a6dd4 再 JMP)
  └─ FUN_00412860  弹管理器每帧遍历(DAT_004a6dac;链表头 mgr+0x70)
        └─ 每颗存活弹 ─► FUN_00411e70  弹每帧 tick
              ├─ FUN_00413860  ★ 字节码 VM(运动脚本,见 §3)
              ├─ 按 obj+0xc68 位场调 10 个行为 handler(见 §4)
              ├─ 位置积分 obj+0xc20 += dt·obj+0xc2c …
              └─ FUN_004124b0  vs 自机碰撞 ─► FUN_004438c0(矩形)/ FUN_004439e0(圆)
                                                  └─ 命中且自机非无敌 ─► FUN_00443f10(自机死亡)
```

- **`FUN_00412860`** ✅:遍历 `DAT_004a6dac` 管理器 `+0x70` 活动链表(节点经 `obj+0x10` 子节点,`node[1]`=next),逐弹调 tick,并按 `obj+0xc84` 分桶进渲染层链(`mgr+0x10+桶*4`),存活计数 `mgr+0x40`。
- **`FUN_00411e70`** ✅:弹每帧 tick——动画计时 → 行为位场派发(§4)→ 位置积分(`+0xc20 += dt·+0xc2c`,dt=`DAT_004a5788`=1.0)→ 碰撞 `FUN_004124b0` → 离屏裁剪(用图形管理器 `DAT_004c0f48` 视野,越界 `FUN_00412670` 回收)。
- **`FUN_004124b0`** ✅:取弹坐标 `+0xc20`、半径 `+0xc40`,据 `obj+0x20` 标志选矩形/圆碰撞;返回 1=命中(置 `+0xc72=3` 转死亡 + 经 `0x40e5c0` 建命中特效)、2=擦弹(`FUN_00444cf0`)、0=未中。

## 2. ★ 对象身份 = 敌弹 ✅(本会话主控一手坐实)

**判据一:接触即伤自机(= 投射物,非敌人)** —— 主控亲反 `0x4438c0`/`0x4439e0`:
- `FUN_004439e0`(圆):`dist²(自机+0x610/+0x614 → 弹)` 比 `(自机判定半径 + 弹半径)²`;自机判定半径取自**解析后的 SHT**(`*(+0x2c788)+4`),弹半径由 XMM2=`obj+0xc40` 传入。**近(dist²<r²)→ 返回 1 + 若自机非无敌(`+0x1663c<1`)调 `FUN_00443f10` 置自机状态 `+0x165a8=4`(死)**;远 → 2(擦弹环)/0(未中)。
- `FUN_004438c0`(矩形):弹 AABB vs 自机判定矩形 `[+0x2c730..+0x2c740]`,重叠 → 同样 `FUN_00443f10`;边缘带 → 2。
- ⚠️ 纠错:身份综合 agent 把 `0x4439e0` 的返回方向读反(以为"远=1");两个对抗 agent + README §A + 本主控一手**三方一致**:**1=命中(近)、2=擦弹、0=未中**,伤害调用就嵌在碰撞函数内。

**判据二:池架构与敌人/自机弹判然不同** —— agent(双源互证)+ 主控抽查一致:
- 弹对象 `0x1478` 字节,**池内嵌**:管理器 `DAT_004a6dac`(`operator_new(0x1403b28)`)里两段 `_eh_vector_constructor` 数组,各 **2001 槽 stride 0x1478**;spawn 从空闲链 `mgr+0x60` pop(`FUN_00412cb0`),**无 per-object `operator_new`**。
- 敌人对象 `0x574c`(~22KB),独立 `operator_new`,进 `DAT_004a6dc0+0x180/+0x184` 链(`FUN_0041aa70`)。弹 opcode 能 spawn 敌人 ≠ 弹是敌人。
- **无自机弹 vs 本池的碰撞 pass**(`FUN_0042d200`〔⚠️**更正**:ExpHP=`GameThread::destructor`,**不是主循环**;真主循环=`Window::do_frame`→`run_all_on_tick`,见 `../shared/th16-main-loop.md`〕里自机管理器 → 本管理器顺序处理,中间无交叉测试)→ 本池**不吃**自机弹伤害 = 弹,非敌(结论不依赖该函数身份,另有伤害链/池架构双证)。
- spawn 时若弹出现在自机判定圈内立即取消(`FUN_00412cb0` 末尾比 `mgr+0x44`)= 经典弹幕行为。

**判据三:行为词汇全是弹运动**(§4):加速/转向/变速变角/反弹/趋点/屏幕夹取;**无** HP/掉落/Boss 阶段等敌人专属逻辑。

> **社区对名**:= ZUN 的 etama / 弹幕弹池;`obj+0xc68` 位场 + `obj+0xc88` 内嵌字节码 = 每颗弹的"弹型/运动脚本"。**近似**:本实现还能 spawn 敌人/子弹,社区 ECL/bullet 文档未记此能力分层,故对名在能力层面是近似(🟡),身份层面=敌弹(✅)。

## 3. ★ 字节码运动 VM `FUN_00413860`(opcode 表)

> **机制/概念解释**(一颗弹为何是携带程序 + 多并发 etEx 效果的微型 VM,而非扁平 struct)见 **`02-bullet-vm-model.md`**。下面是 opcode 速查。

> **形态**(✅ 主控):程序在 `obj+0xc88`,**指令 stride `0x2c`**(11 dword);**PC**=`obj+0xc60`;每帧 `while(PC<0x12)` 顺序执行(≤18 条)。指令字 `[8]`(+0x20)=**opcode**;`[9]`(+0x24)=wait 门。
> **opcode 解码法**(✅ 主控验证):反编译器显示的"float 常量"其位型即整数(`1.4013e-45`=1、`2.8026e-45`=2、`5.6e-45`=4 …)。**关键洞察**(✅):对"行为类"opcode,**opcode 整数值 = 它在 `obj+0xc68` 点亮的位**(如 `c68 |= (uint)opcode`),证实位场标签。高位 opcode 是一次性命令(spawn/置字段),多数不置 c68 位。

> **验证序正确**:VM 由本研究**从零一手反编译**(中立、未看社区表)得出;**之后**用 thcrap 社区的 `ECL-info.md`(etEx 子弹效果表)对照——**不是 priming,是独立外部佐证**(最强一档)。

### 8.1 参数分支二次验证(2 个子 agent + 主控 PE/反汇编复核,2026-06-09)

> 对"角度/速度参数解析分支"逐条核了社区 ECL-info 的细节。**阈值常量全部 PE 实测**(✅):`DAT_00494788=−999990`、`DAT_004946f8=+999990`、`_DAT_004946fc=9999990`、`DAT_004946d8=990`、`DAT_00494778=−990`、`DAT_004946e0=999`。

**社区对 TH16 说对的(✅✅ 一手实证)**:
- **EX_ANGLE(0x10)的 `c` 子模式全中**:`c=0/1/4`→直接设角;`c=2`→`atan2(player−存档位)`(`+0x131c`)+arg;`c=3`→存档角(`+0x1318`)+arg;`c=5/6`→`r·RANDF2`;`c=7`→速度随机`current+s·RANDF2`。目标:角→`+0x1090`、速→`+0x108c`。(地址 `0x413bf0`–`0x413c66`)
- **EX_SAVE 存档机制**:`+0x131c`=存档位、`+0x1318`=存档角、`+0x1314`=存档速,被 c=2/3 回读。
- **EX_VEL(0x40000)** 的 `990/999/−990` 阈值、**EX_VELTIME(0x200000)** 的 `speed=(r−current)/a`(`→+0x1284`)——逐字吻合。
- 角解析基础三分支(`≤−999990`→当前角 / 字面 / `≥999990`→`atan2(player)+m`)+ helper(`0x443840`=朝玩家角、`0x405340/405480/402d30`=归一、`prng_randf_signed`=RANDF2)。

**社区对 TH16 说错的(❌ 实测推翻 —— ECL-info 像是跨版本/后作文档)**:
1. **所谓"7 档角度阶梯"(`1999990/2999990/3999990/4999990` 配 RANDF2、boss 角)在 TH16 根本不存在**:这些常量**全文 .rdata 搜不到**(只有 `999990` 和 `9999990`)。TH16 的 EX_ACCEL/EX_VELTIME 只有**3 分支**(±999990),EX_STEP/EX_ANGLE 多一档 `≥9999990`→存档角(共 4 分支),**无 RANDF2 档、无 boss 角档**。那张 7 档表多半是 **TH17/18 等后作**的行为。
2. **EX_ANGLE c=7 的瞄准阈值**:社区写 `999.0`,实测是 **`990.0`**(`DAT_004946d8`)——社区把 EX_VEL 的 `999` 减数与 c=7 的 `990` 阈值搞混了(俩常量在 .rdata 相邻)。

> **结论**:ECL-info 在**效果代号、c 子模式、存档机制、EX_VEL/VELTIME**上对 TH16 准确;但**角度阶梯的高档与 c=7 阈值是错的/属后作**。**对 TH16 而言我们的 exe 实测此处已比社区文档更准**(已过四闸门:PE 常量实测 + 反汇编分支 + 子 agent 主动找证伪 + 主控独立 PE 复证缺失常量)。ECL-info 仍是优秀指南,但**版本混合,冲突以 TH16 exe 为准**。
