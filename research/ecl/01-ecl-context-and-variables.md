# 01 — ECL 执行上下文结构 + 特殊变量(读取器)— TH16 一手 + Priw8 双证

> **对象**:TH16《鬼形兽》`th16.exe`,imagebase `0x400000`。日期 2026-06-09。
> **方法**:主控一手反编译 `FUN_00424110`(变量读取器)→ 与 **Priw8 ECL 变量表**(`VAR_10` 新引擎基础)逐项交叉验证,**~30/30 吻合**。
> **可信度**:本文 ECL 上下文字段偏移与变量 id↔语义 = **✅✅ 一手 exe + 社区双证**(过四闸门:一手反编译落到具体读取点 + Priw8 独立外部佐证 + 量纲/常识关 + 逐项对名)。仅 TH16(变量集 TH10+ 通用,逐作有增补)。
>
> **这解决了什么**:`00-thecl-format-reference.md` 给了格式侧的"返回寄存器 I3=-9982/F3=-9978";本文在 **exe 里坐实**了它们的运行时存储位置,并顺带解出整个 ECL 上下文(敌机/脚本)结构的一大片字段。**修正**了 `../bullets/01` §7 的一处误判(见 §4)。

## 1. 变量访问器表 `0x4921ac..`(不是 opcode 派发表)

`0x4921b4` 一带是一张 **ECL 变量访问器函数表**,按"访问方式"分槽,均操作 ECL 上下文 `param_1`。各项**只经表被间接调用**(无直接 caller;表基址不以立即数出现在 .text → 被带偏移基址动态索引)。已识别 4 项:

| 函数 | 表槽 | 角色 |
| --- | --- | --- |
| `FUN_00423810` | `0x4921ac` | 同族变体(疑读 int / 另一类) |
| `FUN_00423f80` | `0x4921b0` | **按 id 取变量地址(左值/写)**——只返回可写子集的指针(I0-3=`ctx+0x1498..0x14a4` 等) |
| **`FUN_00424110`** | `0x4921b4` | **按 id 读变量(返回 float10)**——本文主证据,switch on 负 id |
| `FUN_00424c10` | `0x4921b8` | 同族变体(疑写/另一类) |

> ✅ **已定位(更新)**:ECL 主解释循环 = `EclRunContext::ecl_run`(0x472030,见 `04-ecl-vm-interpreter.md`);此变量访问器表经 `ecl_get_int_arg` 的负 id 分支(`vm->vtable` global 路径)调到。本表非 opcode 派发表。

## 4. ★ 修正 `../bullets/01` §7 的假设

旧文档(开放问题 1 / README §B)推测 **`0x4921b4` = ECL opcode 派发表(200+ 项)**。**实测推翻**:
- `0x4921b4` 是 **变量访问器表**(§1),`FUN_00424110` 是其"读特殊变量"项;
- 全文 .text 无立即数引用该表基址,且表内仅 ~十几个 .text 函数指针 + 间隔的非函数槽 → **不是 opcode→handler 扁平数组**。
- 真正的 ECL opcode 解释循环在别处,形态未知(待社区知识 + 继续逆向)。

> 教训(再次):**"agent/二手说读了某偏移" ≠ "用途是 X"**;`FUN_00424110` 的 xref 是 DATA 不代表它是 opcode handler——一手反编译 + 外部对名才定性。已按 `re-overclaim-guard` 复核。

## 5. TH16 特殊变量全表(Priw8,社区权威单源)

新引擎(TH10+)特殊变量按**负 id**寻址;TH16 全集 = `VAR_10` 基础 + 各作增补。核心(VAR_10,FUN_00424110 实现的那批)语义见 §2/§3 表。**TH16 专属新增**:`-9903` = 选定副季(春0/夏1/秋2/冬3/EX 4,只读全局)。其余增补(TH11-15)如 -9959 难度(含 EX=4)、-9908 可击杀敌人数、-9905/-9904(TH15)等。

> **完整逐 id 表**(8/10/11/12/12.5/12.8/13/15/16…逐作)= Priw8 `vars.js`(下方来源)。`access` rw/ro、`scope` l(敌机局部)/g(全局)。**注意 type**:`$`=int、`%`=float——与 thecl sigil 一致。
