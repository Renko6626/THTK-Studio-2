> 方法:Ghidra(ghidra-re MCP)反汇编**用户自有 th16.exe**(32-bit PE,1929 函数) + 本地解析全部
> 9 个 .sht(303 条 shooter 记录) + 多 agent workflow 交叉(parse / 行为回调 / 穷扫网 / 字段图)
> + 对抗复核 + 人工一手复核关键函数。日期 2026-06-08。
> 分级:✅高 / 🟡中 / ❓未解。**仅 TH16(天空璋)**,勿外推他作(TH19 flags 扩到 0x3c,很可能那边才启用)。
> 承接 03(func_* 跳转表)、04(运行时架构)。本篇回答最后一块社区空白:**flags 段(shooter+0x38,
> 0x20 字节 = 16×int16)在 TH16 运行时到底有没有被读、是什么。**

## 0. 一句话结论

**TH16 运行时不读 flags。** 全程序里能读到 shooter 结构的代码已被穷举(9 个 sht 基址消费者 + 全部
16 个 func 指针表项 + 解析器 + 4 个管线 resolver),**读到的最高偏移 = +0x34(func_on_hit 指针),
没有任何一条指令访问 0x38..0x57**。flags 里那些**高度结构化、按角色规律分布**的数据,最合理的解释是
**格式层/编辑器侧配置——被携带、但本二进制从不解引用**,与已证实的"死字段"`_old_on_draw`(+0x30,
解析器修指针却从不调用)同性质。可信度 **HIGH(static-complete 负结论)**;唯一能抬到"绝对"的是动态
内存读断点(需 Windows 跑 exe,我们 headless Linux 做不了)。

## 1. 证据链(发现→推测→验证→结论→证据)

- **发现**:解析器 `sht_parse_resolve_funcptrs@0x443790` 只改 +0x28/2c/30/34(func 指针)、读 +0x00
  (终止符);**整段不碰 +0x38**。spawn 初始化器 `FUN_00444e10` 把 shooter 字段一路灌进运行时槽,
  读到 +0x02/04/08/0c/10/14/18/20/21/22/24/28 就停,**跳过 0x38..0x57**。
- **推测**:flags 要么在某个 func_on_* 行为回调里经 `slot+0xac → 反解 shooter` 被消费,要么在 TH16
  根本是惰性配置。
- **验证**(完整性论证 + 穷举):shooter 结构只活在 .sht blob 里,**唯一取指针路径** =
  `shooter = (idx&0xff)*0x58 + *(sht_base + 0x190 + set*4)`,`sht_base = *(player+0x2c788)`[主] /
  `*(player+0x2c78c)`[副],`player = DAT_004a6ef8`。故任何 flags 读取的数据流上游必然
  **(a) 自己解引用 sht_base 全局并构造 `*0x58 + 表`,或 (b) 接收已解析好的 shooter 指针当参数**。两支全堵:
  - **(a)** 反编译全部 9 个 sht_base 消费者。只有 4 个真正构造逐 shooter 指针:
    `0x444e10 / 0x445360 / 0x445470 / 0x4456d0`,它们读取的 shooter 偏移集合 =
    {0x00,01,02,04,08,0c,10,14,18,20,21,22,24,26,27,28,2c},**最高 +0x2c,无一 ≥0x38**。
    另外 5 个(`0x42f4e0` 读 header+0x08、`0x4440e0` 读 header option_pos +0x40/+0xe8、
    `0x440fb0` 读 header 0x04..0x24 并调解析器、`0x441a50` 仅 free/缓存基址、`0x445a30` 读 header+0x28)
    **都不构造 `*0x58` 逐 shooter 指针**,够不到 +0x38。
  - **全部 16 个 func 指针表项**(对抗复核 dump 三张表得到,见 §3)逐个反编译:其中**3 个**回调真的反解
    shooter——激光 tick `0x446260`(读 +0x14/+0x20)、命中 `0x446e20` 与 `0x446f80`(各只读 +0x02 当伤害,
    传 `FUN_004449b0`);**其余 13 个表项只操作运行时槽,不反解 shooter**。无一读 ≥0x38。
  - **(b)** 4 个 resolver **只拷字段"值"进槽、只长期保存 packed idx**(`slot+0xac` / 池路径
    `piVar9[0x27]`),**从不把 shooter 指针存盘或传给被调者** → 没有被调函数能拿到指针去读 flags。
- **结论**:**flags 0..10、12、14(.sht 里非零的字段)在 TH16 运行时无任何读取点**;0x38..0x57 整段
  无人读。可信度 **HIGH(static-complete)**,版本 **TH16 / 本 th16.exe build**。
- **证据**:`0x443790`(解析器跳过)、`0x444e10`(spawn 最高读 +0x28)、`0x445d40`(命中读 +0x34)、
  `0x446260/0x446e20/0x446f80`(反解 shooter 仅读 +0x14/+0x20/+0x02)、全 16 表项无 ≥0x38;一手复核
  `0x446e20`(本会话亲自反编译确认 `(idx&0xff)*0x58+*(sht_base+400+...)` 后仅 `*(short*)(p+2)`)。

## 3. flags 逐字段:.sht 里怎么分布 × 运行时是否被读

本地解析 303 条记录(pl00=39, pl00sub=20, pl01=39, pl01sub=5, pl02=79, pl02sub=41, pl03=37,
pl03sub=23, pl04sub=20):

- **用到的字段**:flag 0–10、12、14;**恒为 0(padding/保留)**:flag 11(0x4e)、13(0x52)、15(0x56)。
- **结构规律**:偶数字段(0/2/4/6/8/10/12/14)装**ID/值**,紧随的奇数字段装**计数/子参数**,
  成 (value,count) 对;`-1` 在多个偶数字段当**哨兵**。按角色的值常量族很干净:
  - pl00(灵梦/春)= 1 / 257 族;pl01(魔理沙/冬)= 2 / 513 族;pl03(射命丸文/秋)= 5 / 1024 族;
  - pl02(琪露诺/夏)= 1–4 / 101–106 族,**唯一用到 flag9=8**,记录最多(79 条);
  - pl04sub(Extra)最简,只用 flag0/flag2(-1、101–106)。
  - 部分高 option shooter 在 flag0/flag1 存**疑似跨两个 int16 的 packed 32-bit float/角度**
    (如 pl02 [-28033,-16406]、其余 [4059,-16439])。
  → 关联**角色**(值常量)与**火力档/option 阶段**(哪些高字段被填)强于关联季节;同角色主/副弹共享值族。
- **运行时是否被读**:**全部 13 个被用字段 = NO**(见 §1 验证)。每个字段的"猜测角色"都只是**数据形状
  推断、零代码支撑**,故**不写成定论**。

## 4. 对抗:flags 到底有没有运行时含义

- **支持"有含义"**:数据高度结构化(value/count 对、按角色常量族、`-1` 哨兵、16xxx 位掩码状值、
  随火力档增长),flag9 仅 pl02 触碰——像活配置。
- **支持"TH16 实际不用"**:**没有代码读它**(§1 闭合);TH16 明确携带死字段(`_old_on_draw` 修指针却
  从不调用)即先例;"结构化"完全可由**格式层/编辑器侧数据**解释(可能被别的 ZUN 工具或早期作消费),
  **结构化 ≠ 被读**。
- **判决**:证据favor**"本 TH16 build 无运行时读者"**。负结论建立在完整性闭合上(由 6+ 个 resolver 一致
  佐证:都只存 packed idx、不存指针)。残留不确定性已降到只剩动态层面。

## 4b. 意外收获:SHT header +0x04 = 玩家判定点半径 + 命中/擦弹判定函数 ✅

兜底扫抓到的"漏网函数" `0x4439e0` 一手追下去,多解出一个 **SHT header 字段**和一处核心判定:

- **`sht_base + 0x04`(.sht header 的 float)= 玩家「判定点(hitbox)死亡半径」**。focus 时(player
  `+0x1664c & 0x10`)按 `player+0x2c7c8/2c7cc`(focus 缩放)收缩——与低速判定点缩小的表现一致。
- **`FUN_004439e0` = 玩家命中判定(+近距环)**(`(玩家坐标 → 目标点)` 距离² 比半径):

## 5. 对 IDE 的指导(回流 ../docs/)

- TH16 shooter 字段 **0x00–0x37 已全解**(见 §2)+ header 字段图(见 §2b),可作 SHT 编辑器字段定义层。
- **「编辑无效」字段必须标注(否则误导用户)**:这些字段在 .sht 里有值,但 TH16 运行时**不采纳**——
  - header `+0x04/+0x08/+0x0c`(判定/擦弹/道具半径):init 被 **exe 内按角色硬表覆写** → 改 .sht 无效,
    玩家判定半径**不可经 SHT 编辑**;`+0x24`:覆写为常量 100;
  - shooter `flags`(0x38–0x57)、`_old_on_draw`(+0x30):无人读;
  - 编辑器应把它们标为**只读/"运行时未采纳"**,但**字节级原样保留**(格式层数据,改了可能破坏跨工具兼容)。
- **真数据驱动(改 .sht 有效)**:移速 `+0x10–0x1c`、`pwr_lvl_cnt +0x20`、`max_dmg +0x28`、以及全部
  shooter 几何/计时/func/option 字段——这些才是 SHT 编辑器该开放编辑的。
- **版本边界**:本结论**仅 TH16**;TH19 flags = 0x3c、按角色硬表也可能不同,需各自反汇编。带版本元数据。

## 6. 全二进制兜底扫(已执行,✅确认负结论)

`scripts/scan_flags_readers.py`(指令层全程序扫,148712 条指令 / 1928 函数;`run.sh` 跑,~40s)。
核心洞察:**任何能解析 shooter 指针的函数,指令里必然出现 sht_base 立即数 `0x2c788`/`0x2c78c`**。

- **引用 sht_base 的全部函数 = 14 个**(= shooter 指针生产者完整集):
  `0x42f4e0, 0x440fb0, 0x441a50, 0x4439e0, 0x4440e0, 0x444e10, 0x445360, 0x445470, 0x4456d0,
  0x445a30, 0x445d40, 0x446260, 0x446e20, 0x446f80`。其中 **13 个本研究已分析**;
  **`0x4439e0` 是兜底扫新捕获、之前所有 agent 漏掉的**——一手反编译确认它读 `*(sht_base+0x04)`,
  **无 `*0x58`/无 +0x190 表,不解析逐 shooter,不读 +0x38..0x57** → header 消费者,安全。**(网有效且
  结果干净。)** 顺带解出**意外收获**(见下),不影响 flags 负结论。
- **这 14 个 gate 函数里 disp∈[0x38,0x57] 的 [reg+disp] 访问共 68 处,经核对 base 寄存器全是
  运行时 slot / 子弹池对象 / header / 其它结构,无一是 shooter 指针**(与逐个反编译一致:shooter 指针在
  这些函数里只被读 +0x02/+0x14/+0x20/+0x34)。全程序此类 disp 访问共 1938 处,绝大多数属别的结构。
- **交叉核**:全程序同时含立即数 `0x58` 与 `400(0x190)` 的函数 17 个;除 8 个∈gate 的解析器 + 解析器
  `0x443790`(从**参数**拿 base,已知不读 flags)外,其余 8 个**不引用 sht_base**,够不到玩家 shooter。

> **判定:negative 在全二进制静态层确认。** 14 个 gate 函数(唯一能解析 shooter 的集合)无一读
> 0x38..0x57;唯一从参数拿 base 的解析器也不读。可信度 **HIGH(static-complete)**。
> 残留只剩动态层(下),且无任何静态线索指向存在读者。
