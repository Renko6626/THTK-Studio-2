# 糗事档 99:TH16 的"可配置判定半径"其实是颗哑弹

> 一则有趣的逆向小事,**双重糗**:一半是 ZUN 的(留了会读的管线却把数据废掉),一半是我们的
> (把"会被读"当成了"是活的")。正经结论在 `05-th16-flags-no-runtime-read.md` §2b/§4b,这里只讲故事 + 教训。
> 可信度:TH16 覆写 = ✅一手验证;"早期作真从 .sht 读半径" = 🟡 强推断(未 RE 过 TH13–15)。

## 一句话

SHT 格式里有个**可配置的玩家判定半径**字段(header `+0x04` = sht-webedit 的 `hitbox_u`),**碰撞代码也
确实会去读它**——但 TH16 在加载后用一行赋值把它盖成硬编码的 `3.0`,于是你在 .sht 里填的任何值都是**哑弹**。

## 现场(一手反汇编,th16.exe)

1. **读取路径是真的、活的**:玩家命中/擦弹判定 `FUN_004439e0` 每次碰撞都读 `*(sht_base + 0x04)` 当判定半径
   (`0x443A1C MOVSS XMM1,[EAX+0x4]`,喂进距离²比较)。从这看,它**就是个数据驱动的判定半径**。
2. **但 init 把它 stomp 了**:`player_shot_init @0x440fb0` 在解析 .sht 之后:
   ```c
   *(undefined4 *)(*piVar2 + 4) = *(undefined4 *)(&DAT_00492c98 + DAT_004a57a4 * 4);
   //   sht_base+0x04(.sht 的可配置半径)  =  exe 内按角色常量表[char] = 3.0
   ```
3. **时序决定一切**:解析器不碰 +0x04 → init **覆写成 3.0** → 碰撞才去读 → 读到的永远是 3.0。
   文件值从"读进内存"到"被覆写"之间,**没有任何代码用过它** → 纯哑弹。实测 4 角色常量表全是 `3.0`。

## 糗一:ZUN 这操作不太优雅

留着一整套"读可配置半径"的管线(格式有字段、编辑器认、碰撞真读),然后**不删读取、改用一行常量把数据
盖掉**。结果是个**长得像可配置、其实是哑弹**的字段,专坑读格式/写工具/做逆向的人。

为什么不直接删读取?——只能合理推断(🟡):SHT 是跨作共用格式 + 有外部编辑器(sht-webedit),
**stomp 一行比改格式/改 loader/改每作省事**;而且把判定半径集中到一张 exe 常量表,调平衡只改一处。
属于 ZUN 一贯的"务实丑"(ReC98 扒早期作满是这种魔法数字/能跑就行),**按 clean code 不优雅,但在他约束下是理性取舍**。

## 糗二:我们(我)也被它骗了

我一开始一口咬定 **"+0x04 = 从 .sht 读的判定半径"**——"它会被读"这半句是**真的**,"它是活的数据"那半句是
**假的**。直到全二进制兜底扫(`scan_flags_readers.py`)把漏网的 `player_shot_init` 抓出来,才看见那行 stomp。
> **教训:会读 ≠ 数据驱动。** 一个字段"有读取点"只说明读取路径存在,不代表它的值没在别处被覆写/伪造。
> 要确认"可配置",得顺数据流查它**进内存后到被读之间有没有被改写**。详见纪律 `00-METHOD-*` 与
> memory `re-overclaim-guard`。

## 为什么敢说"之前确实会读"(但只敢标 🟡)

最硬的论据其实就藏在代码里:**你只会去 stomp 一个否则会被用到的字段**。0x4439e0 的读取路径是现成、通用的,
ZUN 没删它、而是喂常量——这强烈暗示这套读取是**引擎血统里"本来吃这个字段"的遗留**,TH16 只是"保留读取、
掐断数据"。再配合 Touhou Wiki:**TH13–15 灵梦判定 2.0、其余 3.0**(有角色差异),到 TH16 统一成 3.0。
→ 合理推断:这字段在更早的引擎里曾是真·数据驱动。**但我们没反汇编过 TH13–15,故标 🟡**;要坐实,
得对那几作的 exe 同样查"读取 vs 覆写"。

## 给 IDE 的启示

SHT 编辑器**不能拿"字段存在/被读"当"改了有效"**。判定/擦弹/道具半径在 TH16 是 exe 硬编码,
编辑器应标注"**运行时不采纳**"(改了不生效),但字节级保留。判定字段的"可编辑性"必须**逐作**按
"读取后是否被覆写"来定——这正是这则糗事的实操价值。
