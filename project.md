
# THTK-Studio 开发计划

## 1. 项目目标

THTK-Studio 是一个面向东方 Project 原作脚本魔改的专用 IDE。
它的目标不是做一个普通文本编辑器，而是做一个围绕 **ECL / ANM / MSG / STD** 等脚本与资源工作流的桌面开发环境。

核心目标如下：

1. 集成现有 Touhou 脚本工具链，减少手动解包、编辑、打包的繁琐流程
2. 提供比普通文本编辑器更强的脚本编辑体验
3. 为东方脚本开发提供专用的语义辅助能力
4. 提供 sprite、动画、时间线等可视化预览
5. 为未来的 AI 辅助补全、脚本解释、自动重构预留接口

## 2. 总体技术路线

## 2.1 技术栈

### 桌面框架

* **Tauri v2**

### 前端

* **Vue 3**
* **TypeScript**
* **Monaco Editor**
* 组件库可选：Naive UI

### 后端 / 宿主

* **Rust**
* Tauri command
* 必要时使用 sidecar / 后台服务进程

### 外部工具链

* **thtk** 作为现阶段主要解包 / 打包 / 编译 / 反编译工具
* 后续可考虑部分能力替换为自研 Rust 实现

### 关于thtk

thtk一共有五个核心工具
- thdat 负责把游戏的.dat文件解包成若干.ecl .anm等分开的二进制文件，以及把修改后的文件重新打包成.dat
- thecl 负责解包.ecl文件，ecl文件控制游戏主流程和弹幕的射击，把它们转换成文本格式，或者把修改后的文本重新编译成.ecl
- thmsg 负责解包.msg文件，msg文件控制对话和文本显示，把它们转换成文本格式，或者把修改后的文本重新编译成.msg
- thanm 负责解包.anm文件，anm文件控制精灵素材和动画播放，把它们转换成图像素材与文本格式的控制脚本，或者把修改后的文本/图像素材重新编译打包成.anm
- thstd 负责解包.std文件，std文件控制3d背景的贴图和动画，把它们转换成文本格式，或者把修改后的文本重新编译成.std

## 3. 系统分层设计

项目建议分为四层：

## 3.1 UI 层

负责窗口、编辑器、面板、菜单、文件树、预览区等界面展示。

## 3.2 编辑器集成层

负责 Monaco 的语言支持接入，包括高亮、补全、跳转、报错显示等。

## 3.3 领域服务层

负责脚本解析、符号索引、引用关系、资源引用、时间线模型等领域逻辑。

## 3.4 工具链与系统层

负责文件系统、项目管理、进程调用、thtk 封装、缓存、配置管理等。

# 4.2.1 内嵌终端能力

## 功能目标

提供一个真正可用于项目命令执行与交互式 CLI 的内嵌终端，而不是只做一次性命令调用面板。

## 需要实现的内容

* 持续 shell 会话
* 终端输入输出流式通信
* 支持 `pwsh` / `cmd`
* 支持工作目录切换
* 支持终端尺寸变化
* 支持 ANSI 控制序列
* 支持交互式命令和 TUI 程序

## 前端负责

* 终端面板 UI
* 终端会话管理
* 终端输入转发
* 终端输出渲染
* 与编辑器布局集成

## 后端负责

* PTY / ConPTY 会话创建与销毁
* shell 进程生命周期管理
* 输入写入与输出读取
* 终端 resize
* 会话状态维护
* 将输出通过事件流推送给前端

## 当前实现说明

当前项目里已经有一个“内嵌命令面板”，但它本质上只是：

* 输入一条命令
* Rust 启动一次子进程
* 收集输出后返回

这不是真正的终端。
它不能正确承载以下场景：

* `codex`
* `claude code`
* `vim`
* Python / Node REPL
* 长时间运行并持续交互的命令

因此后续如果要支持真正终端，技术路线应该明确为：

* 前端使用 `xterm.js` 或同类终端渲染器
* Rust 侧实现 PTY，Windows 下优先考虑 `ConPTY`
* 前后端之间通过流式事件通信，而不是一次 `invoke` 返回整段文本

不要继续在“一次性命令执行 command”模型上追加交互式终端能力。

## 5.1 主要职责

### 具体包括

* 窗口布局
* 文件树 / 项目树
* Monaco 编辑器集成
* 标签页管理
* 面板系统
* 日志 / 问题 / 搜索结果展示
* 预览界面
* 设置界面
* 编辑器与预览联动

## 5.2 前端不应负责

* 复杂脚本解析

* 外部工具流程编排
* 大规模文件扫描
* 长时间阻塞任务

## 6. 后端 / Rust 宿主职责总表

## 6.1 主要职责

后端负责“本地能力、性能敏感逻辑、领域逻辑和工具链封装”。

### 具体包括

* 文件系统访问
* 项目配置管理
* 工具链调用
* 日志采集
* 脚本解析
* 诊断生成
* 符号索引
* 全局缓存
* 预览数据构建
* 后台任务调度

## 6.2 后端设计原则

* 返回结构化结果，不返回杂乱文本
* 长任务不能阻塞 UI
* 错误必须明确可追踪

* 外部工具必须经过封装，不直接散落调用

## 8. 当前最重要的实施重点

当前最值得优先落实的不是所有高级功能，而是以下四件事：

1. **确定项目结构和配置文件格式**
2. **完成 thtk 工作流集成**
3. **完成编辑器与 Rust 宿主之间的清晰边界**
4. **区分命令执行面板与真正 PTY 终端，避免把临时方案演变成长期架构**

如果这几件事先做好，后续语言服务、预览器和 AI 都能自然接入。
如果这些事没做好，后面功能越多，架构越容易失控。

---

## 9. 结论

THTK-Studio 的本质不是“桌面文本编辑器”，而是一个面向东方脚本生态的专用开发环境。
因此开发时应始终围绕以下原则：

* 先保证工作流闭环
* 再构建语言服务
* 再做可视化预览
* 最后再接 AI

前端负责体验与展示，后端负责性能、语义和工具链。
两边边界清晰，项目才会长期可维护。
